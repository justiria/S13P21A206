# 🧰 Git 명령어 핵심 정리

협업 중 자주 사용하는 Git 명령어와 충돌 해결, 시각화 도구에 대해 정리한 문서입니다.

---

## 🔀 `git merge` – 브랜치 병합

- **Current Change**: 현재 브랜치 코드 (HEAD)
- **Incoming Change**: 병합 대상 브랜치의 코드
- **충돌 시**: 반드시 팀원과 상의하여 해결!

> 💡 **Merge Request(MR)**  
> - 병합 시 모든 팀원에게 변경 상황 알림  
> - GitLab ↔ Mattermost 연동 시 자동 알림

---

## 📚 `git rebase` – 커밋 히스토리 정리

- 커밋 기록을 깔끔하게 정리하여 메인 브랜치에 반영
- **주의**: 히스토리가 변경되므로 공동 작업 중엔 사용 지양
- **추천 사용 시점**: Push 전, 로컬에서만 사용

---

## ⏪ `git revert` – 커밋 취소 (안전한 방식)

- 기존 커밋을 되돌리는 **새 커밋을 생성**
- 기존 히스토리는 유지되어 복구 가능

```bash
$ git revert HEAD           # 최근 커밋 취소
$ git revert [커밋 해시]    # 특정 커밋 취소
⚠️ 여러 커밋이 상호 연관되어 있으면 충돌 위험 존재

💥 git reset – 커밋 삭제 (위험!)
커밋 히스토리를 직접 삭제

다음 3가지 옵션 존재:

옵션	설명
--soft	커밋만 삭제 (작업 파일은 유지)
--mixed	커밋 + staging 삭제
--hard	모든 변경사항 삭제 (복구 불가!)

🚨 웬만하면 사용하지 말고 revert를 사용하세요!

📌 git status – 현재 상태 확인
현재 브랜치, 커밋 여부, staging 여부 확인

작업 전 필수 확인 습관화!

🗃️ git stash – 작업 임시 저장
현재 작업 내용을 잠시 보관하고, 브랜치를 전환하거나 테스트 가능

bash
복사
편집
$ git stash        # 임시 저장
$ git stash pop    # 다시 꺼내기
🌳 git log --oneline --graph --all – 브랜치 시각화
전체 브랜치의 커밋 흐름 시각화 가능

브랜치 분기, 병합 흐름을 직관적으로 파악 가능

🖥️ Git GUI 툴 추천
🟦 GitHub Desktop
✅ 초보자 친화적, 깔끔한 UI

❌ 복잡한 브랜치 작업, 고급 기능 부족

🌲 SourceTree
✅ 브랜치 시각화 훌륭, 고급 기능 지원 (rebase, stash 등)

❌ UI가 복잡하고 실수 위험 있음

🧩 VS Code Git 확장 (GitLens, Git Graph 등)
✅ VS Code 내에서 바로 Git 작업 가능

✅ Diff 및 충돌 해결 간편

❌ 전체 브랜치 흐름 파악은 다소 어려움

✅ Git 실전 사용 체크리스트
작업 전: git status로 현재 상태 확인

명령어 입력 전: 이 명령이 되돌릴 수 있는가 반드시 체크

확실하지 않다면? → 팀원과 상의

🧠 Git은 '혼자 잘 쓰는 도구'가 아닌 '함께 잘 쓰는 협업 도구'입니다.
