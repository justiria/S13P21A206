🛡️ 세션 인증 vs 토큰 인증
웹 기반 서비스를 제작할 때 사용자 인증은 기본이자 핵심 기능입니다. 사용자 인증을 구현하기 위한 대표적인 방식으로 세션(Session) 기반 인증과 토큰(Token) 기반 인증이 있으며, 서비스의 성격과 요구사항에 따라 적절한 인증 방식을 선택하게 됩니다.

✅ 세션 인증 (Session-based Authentication)
🔹 특징
MPA(Multiple Page Application) 기반, 전통적인 구조

Server Side Rendering(SSR) 방식

서버의 메모리나 클라이언트 쿠키를 통해 인증 유지

대규모 시스템에서 성능·보안을 위한 전략적 선택

전통적인 웹 언어: PHP, JSP, ASP 등에서 많이 사용

✅ 장점
서버 측에서 사용자 인증 확인이 명확하고 용이

세션 정보를 서버에서 직접 관리 → 변조 위험 낮음

부가정보가 적어 Client ↔ Server 간 송수신 데이터 적음

데이터 양이 상대적으로 적음

❌ 단점
서비스 Scale-Up 시 서버 메모리 부하 문제

별도 메모리 관리 시스템(Redis, Memcached) 필요

도메인 기반 쿠키 한정 → 서브 도메인 관리 번거로움

✅ 토큰 인증 (Token-based Authentication)
🔹 특징
SPA(Single Page Application) 기반, 최신 트렌드

사용자 경험 중심의 복잡한 Front-end와 적합

Vue, React, Angular 등에서 주로 활용

다양한 디바이스(웹, 모바일, IoT 등) 대응 가능

✅ 장점
멀티 플랫폼 대응 유리

인증 정보를 클라이언트가 보유 → 서버 부담 감소

서버의 Scale-Up 부담이 적음

세션 관리에 대한 서버 메모리 소모 감소

도메인 제약 없음 → 다양한 도메인에서 사용 용이

❌ 단점
클라이언트 저장 토큰 탈취 시 보안에 취약

Signature 등 부가정보로 인해 데이터 전송량 증가

인증 시 DB 조회 발생 → 성능 부하 가능성

🔍 선택 기준
기준	세션 인증	토큰 인증
시스템 구조	전통적 MPA	현대적 SPA
보안	서버 중심 → 상대적으로 안전	클라이언트 저장 → 탈취 위험
서버 부하	세션 저장으로 부담 ↑	서버 부담 ↓
확장성	제한적 (단일 도메인)	유연 (다양한 플랫폼 대응)
데이터 전송량	적음	많음

📌 결론
내부망 서비스, 보안이 중요한 전통적 시스템 → 세션 인증이 적합

모바일/SPA 기반 서비스, 다양한 플랫폼과 도메인 대응이 필요한 경우 → 토큰 인증이 유리

서비스의 구조와 사용 환경, 요구사항을 고려해 두 방식 중 적절한 방법을 선택하는 것이 중요합니다.
