# Jira

## 1. Jira 소개

**Jira**는 Atlassian사에서 개발한 **프로젝트 관리 및 이슈 추적 도구**입니다. 원래 이름은 고질라의 일본식 이름인 '고지라(Gojira)'에서 유래했으며, 버그 추적 시스템인 '버그질라(Bugzilla)'를 대체하겠다는 포부를 담고 있습니다.

초기에는 소프트웨어 개발팀의 버그 추적을 위해 주로 사용되었지만, 현재는 애자일 방법론을 완벽하게 지원하고 다양한 비즈니스 환경에 맞춰 기능을 확장하면서 **IT, 마케팅, 인사, 재무 등 모든 팀의 협업과 업무 관리를 위한 핵심 툴**로 자리 잡았습니다.

**Jira를 사용하는 이유:**

- **업무의 투명성:** 모든 업무가 '이슈' 단위로 등록되고 추적되므로, 누가 어떤 일을 하고 있으며 현재 어떤 단계에 있는지 명확하게 파악할 수 있습니다.
- **효율적인 프로젝트 관리:** 애자일(스크럼, 칸반) 방법론을 기반으로 프로젝트 계획, 실행, 검토 과정을 체계적으로 관리할 수 있습니다.
- **강력한 커스터마이징:** 우리 팀의 업무 방식에 맞게 워크플로우, 필드, 이슈 타입 등을 자유롭게 변경할 수 있습니다.
- **우수한 확장성:** Confluence(문서 협업), Bitbucket(코드 형상 관리) 등 다른 Atlassian 제품 및 수많은 서드파티 앱과 연동하여 기능을 무한히 확장할 수 있습니다.

## 2. Jira 핵심 용어

### 프로젝트 (Project)

Jira에서 **프로젝트**는 특정 목표를 가진 작업들의 집합을 담는 가장 큰 단위입니다. 소프트웨어 개발, 마케팅 캠페인, 신규 직원 온보딩 등 모든 것이 하나의 프로젝트가 될 수 있습니다.

Jira에는 크게 두 가지 유형의 프로젝트가 있습니다.

- **회사 관리형 (Company-managed):** 관리자가 전체 조직의 표준에 맞춰 워크플로우, 권한 등을 중앙에서 관리합니다. 표준화된 프로세스가 중요한 대규모 팀에 적합합니다.
- **팀 관리형 (Team-managed):** 팀이 독립적으로 자신들만의 작업 방식과 설정을 자율적으로 관리합니다. 빠르고 간단한 설정으로 시작하고 싶은 소규모 자율 팀에 적합합니다.

### 이슈 (Issue)와 계층 구조

**이슈**는 Jira에서 추적하는 모든 종류의 작업 단위를 의미합니다. 버그, 신규 기능, 사용자 스토리, 단순한 할 일(Task) 등 모든 것이 이슈가 될 수 있습니다. Jira는 업무를 체계적으로 관리하기 위해 다음과 같은 계층 구조를 사용합니다.

- **에픽 (Epic):** 거대한 기능이나 목표를 나타내는 최상위 이슈입니다. '2025년 3분기 신제품 출시'나 '사용자 인증 시스템 개편'과 같이 여러 스프린트에 걸쳐 진행되는 큰 규모의 작업을 에픽으로 설정합니다.
  - **스토리 (Story) / 태스크 (Task):** 에픽을 구성하는 구체적인 작업 단위입니다.
    - **스토리:** 사용자의 관점에서 작성된 기능 요구사항입니다. (예: "사용자로서, 이메일과 비밀번호로 로그인할 수 있다.")
    - **태스크:** 사용자와 직접적인 관련이 없는 기술적인 작업입니다. (예: "로그인 API 서버 구축")
    - **버그 (Bug):** 소프트웨어의 결함을 나타내는 이슈입니다.
    - **서브태스크 (Sub-task):** 하나의 스토리나 태스크를 더 작은 단위로 나눈 것입니다. (예: '로그인 UI 디자인', '로그인 API 개발', '로그인 기능 테스트')

이를 시각화하면 다음과 같습니다.

| 이슈 유형             | 역할 및 목적                           | 구체적인 예시                                  | 관계 (상위/하위)                                                   |
| --------------------- | -------------------------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| 에픽 (Epic)           | 여러 스프린트에 걸친 큰 기능 또는 목표 | "새로운 모바일 앱 결제 기능 출시"              | 최상위 작업 단위. 여러 스토리, 태스크, 버그를 포함.                |
| 스토리 (Story)        | 사용자 관점의 기능 요구사항            | "사용자로서, 신용카드로 상품을 결제하고 싶다." | 에픽의 하위 이슈. 여러 서브태스크를 가질 수 있음.                  |
| 태스크 (Task)         | 특정 작업 또는 해야 할 일              | "결제 API 서버 환경 구축"                      | 에픽의 하위 이슈 또는 독립적 존재. 여러 서브태스크를 가질 수 있음. |
| 서브태스크 (Sub-task) | 스토리나 태스크를 구성하는 세부 작업   | "결제 버튼 UI 디자인", "카드사 연동 모듈 개발" | 스토리 또는 태스크의 하위 이슈. 가장 작은 작업 단위.               |

```
Epic: 사용자 인증 시스템 개편
└─ Story: 이메일 로그인 기능 구현
├─ Sub-task: 로그인 페이지 UI 디자인
├─ Sub-task: 로그인 API 개발
└─ Sub-task: 로그인 기능 테스트
└─ Story: 소셜 로그인 기능 구현
├─ Sub-task: 구글 로그인 연동
└─ Sub-task: 카카오 로그인 연동
```

### 워크플로우 (Workflow)

**워크플로우**는 하나의 이슈가 생성부터 완료까지 거치는 과정을 정의한 것입니다. 가장 단순한 워크플로우는 `할 일 (To Do) -> 진행 중 (In Progress) -> 완료 (Done)`로 구성됩니다.

팀의 필요에 따라 `코드 리뷰 (Code Review)`, `QA 테스트`, `배포 대기` 등 다양한 상태(Status)와 상태 간의 이동 규칙(Transition)을 추가하여 맞춤형 프로세스를 설계할 수 있습니다.

### 보드 (Board)

**보드**는 워크플로우를 시각적으로 표현한 공간입니다. 팀원들은 보드를 보며 현재 진행 중인 모든 작업을 한눈에 파악하고, 드래그 앤 드롭 방식으로 이슈의 상태를 쉽게 변경할 수 있습니다. Jira는 주로 두 가지 종류의 보드를 제공합니다.

- **스크럼 (Scrum) 보드:**

  - **목적:** 정해진 기간(스프린트) 동안 정해진 양의 작업을 완료하는 데 초점을 맞춥니다.
  - **특징:** 백로그, 스프린트 계획, 번다운 차트 등의 기능이 포함됩니다.
  - **적합한 팀:** 주기적인 계획과 리뷰를 통해 제품을 점진적으로 개발하는 소프트웨어 개발팀.

- **칸반 (Kanban) 보드:**
  - **목적:** 업무의 흐름을 지속적으로 유지하고 병목 현상을 최소화하는 데 초점을 맞춥니다.
  - **특징:** 진행 중인 작업의 수(WIP, Work In Progress)를 제한하는 기능이 중요합니다.
  - **적합한 팀:** IT 지원, 마케팅, 운영과 같이 업무 요청이 지속적으로 들어오고 우선순위에 따라 처리해야 하는 팀.

## 3. Jira 시작

### 1단계: 프로젝트 생성하기

1.  Jira 상단 메뉴에서 **프로젝트 > 프로젝트 만들기**를 클릭합니다.
2.  팀의 목적에 맞는 템플릿을 선택합니다. (예: 소프트웨어 개발을 위한 '스크럼' 또는 '칸반')
3.  **팀 관리형**과 **회사 관리형** 중 하나를 선택합니다. (처음이라면 '팀 관리형'이 더 간단합니다.)
4.  프로젝트 이름과 키(Key)를 지정하면 프로젝트가 생성됩니다. (키는 이슈 번호의 접두사로 사용됩니다. 예: `TECH-123`)

### 2단계: 이슈 생성 및 관리하기

1.  상단 메뉴의 **만들기** 버튼을 클릭합니다.
2.  **프로젝트**와 **이슈 유형**(스토리, 태스크, 버그 등)을 선택합니다.
3.  **요약(제목)**, **설명(상세 내용)**, **담당자**, **우선순위** 등 필요한 필드를 채웁니다.
4.  '만들기'를 클릭하면 이슈가 생성되어 백로그나 보드에 추가됩니다.

### 3단계: 애자일 보드로 업무 흐름 시각화하기

- 프로젝트 좌측 메뉴에서 **보드**를 클릭합니다.
- `할 일` 열에 있는 이슈를 `진행 중` 열로 드래그 앤 드롭하여 작업 시작을 알립니다.
- 작업이 완료되면 `완료` 열로 이동시킵니다.
- 보드의 각 열은 워크플로우의 상태와 1:1로 매핑됩니다.

### 4단계: 백로그 관리와 스프린트 계획하기 (스크럼 프로젝트)

1.  **백로그 관리 (Backlog Grooming):**
    - 좌측 메뉴에서 **백로그**를 클릭합니다.
    - 백로그 목록에서 이슈들을 드래그하여 우선순위를 조정합니다.
    - 중요하고 긴급한 이슈를 목록의 맨 위로 올립니다.
2.  **스프린트 계획 (Sprint Planning):**
    - 백로그 상단에 있는 '스프린트' 영역으로 우선순위가 높은 이슈들을 드래그 앤 드롭하여 이번 스프린트에 할당합니다.
    - 팀의 작업량을 고려하여 적절한 양의 이슈를 할당합니다.
3.  **스프린트 시작:**
    - **스프린트 시작** 버튼을 클릭하고 스프린트 기간(보통 1~2주)과 목표를 설정합니다.
    - 스프린트가 시작되면 해당 이슈들이 '스크럼 보드'에 나타나며, 팀은 이제 보드를 보며 작업을 진행합니다.

## 4. Jira 고급 기능 및 연동

### 워크플로우 커스터마이징

기본 워크플로우(`To Do -> In Progress -> Done`)가 팀에 맞지 않는다면 직접 수정할 수 있습니다.

1.  **프로젝트 설정 > 워크플로우**로 이동합니다.
2.  워크플로우 편집기에서 **상태 추가** 버튼으로 '코드 리뷰', 'QA 대기' 같은 새로운 단계를 추가합니다.
3.  상태와 상태를 잇는 **전환(Transition) 추가**를 통해 이슈가 어떤 순서로 이동할지 규칙을 정의합니다. (예: `In Progress` 상태에서만 `Code Review` 상태로 이동 가능)

### JQL (Jira Query Language)

JQL은 SQL과 유사한 Jira만의 강력한 검색 언어입니다. 기본 검색으로는 찾기 힘든 특정 조건의 이슈를 정확하게 찾아낼 수 있습니다.

- 상단 검색창에서 **이슈 및 필터 > 고급 이슈 검색**으로 이동합니다.
- JQL 입력창에 쿼리를 작성합니다.

**유용한 JQL 예시:**

- `project = "TECH" AND assignee = currentUser() AND status = "In Progress"`
  - 'TECH' 프로젝트에서, 현재 로그인한 나에게 할당된, '진행 중' 상태인 모든 이슈
- `priority = Highest AND created >= -7d`
  - 가장 높은 우선순위를 가지며, 최근 7일 이내에 생성된 모든 이슈
- `fixVersion = "2.0 Release" AND status changed to "Done" during (startOfWeek(), endOfWeek())`
  - '2.0 릴리즈' 버전에 포함되며, 이번 주에 '완료' 상태로 변경된 모든 이슈

### Confluence & Bitbucket

- **Jira + Confluence (문서 협업 도구):**

  - **요구사항 관리:** Confluence에 제품 요구사항 명세서(PRD)를 작성하고, 관련 Jira 에픽과 스토리를 페이지에 직접 링크하거나 생성할 수 있습니다.
  - **지식 베이스:** 프로젝트 기획서, 회의록, 기술 아키텍처 문서 등을 Confluence에 체계적으로 정리하고 Jira 이슈와 연결하여 컨텍스트를 제공합니다.
  - **보고서 작성:** Confluence 페이지에 Jira 이슈 현황을 실시간으로 보여주는 차트나 필터링된 목록을 삽입하여 동적인 프로젝트 보고서를 만들 수 있습니다.

- **Jira + Bitbucket (Git 코드 관리 도구):**
  - **개발 현황 추적:** Jira 이슈 키를 포함하여 브랜치를 생성하고(`git checkout -b TECH-123-login-feature`), 커밋 메시지에 키를 남기면, 해당 Jira 이슈 내에서 관련 브랜치, 커밋, Pull Request 현황을 바로 확인할 수 있습니다.
  - **자동화:** 특정 브랜치가 머지되거나 Pull Request가 승인되면, 연결된 Jira 이슈의 상태를 자동으로 '완료'로 변경하는 등의 자동화 규칙을 설정할 수 있습니다.
