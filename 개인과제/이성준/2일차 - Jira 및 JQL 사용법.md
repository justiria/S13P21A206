# 2일차 - Jira 및 JQL 사용법

날짜: 2025년 7월 15일

### 왜 Jira를 사용할까?

- Issue Tracking : Jira는 Issue Tracking 툴이다. 이슈 트래킹은 개인과 팀의 협업(issue)의 상황과 상태를 지속적으로 공유하고, 팀의 흐름을 파악하기 위한 도구이다. → 수 많은 이슈 트래킹 툴이 있으나 주로 Jira를 사용한다.
- 그러면 Gitlab과 같은 형상 관리 도구에서도 이슈를 확인할 수 있는데, 왜 굳이 Jira를 사용하는가? 이는 프로젝트 매니지먼트 측면에서 이유를 확인할 수 있다. 대시보드와 백로그등의 기능을 통해서 이슈를 효율적으로 관리할 수 있다.
- Agile의 등장 : 워터폴과 같은 고전적인 개발주기가 아닌 즉각적인 대응이 가능한 프로그램 등장
    - Scrum
    - Kanban : 일본어로 메모를 의미하며 진행 상황을 공유하고 이를 옮기는 것. 이를 Jira에서는 포스트잇과 같은 형태가 아닌 온라인에 대신 올리는 것이 가능하다
    - Scrum Meeting : 5-10분 사이에 저번에 한 일과 오늘 할일을 간단하게 짚고 넘어가는 것
    - Backlog : 개발을 해야하는 내용을 적어둔것
    - DevOps : 개발과 운영팀을 하나로 합쳐서 운영. 즉, 두 팀이 하나로 합쳐져서 업무를 도와 효율성을 높임
        - 잘 수행하려면? 반복적인 작업을 Tool을 이용하여 자동화, 팀원 모두가 알고 있는 하나의 공유된 지표가 필요, 장애나 이슈가 있을 때 이를 바로 팀원들가 공유헐 수 있다.
        - (중요) DevOPs의 모든 단계에 유일하게 JIRA가 모든 DevOps 단계에 포함
- SRE : Site Reliaiblity Engineering
- Atlassian ROVO

### Jira & JQL 활용법

→ 일단 이슈를 생성해야한다.

- 스토리, 작업, 버그, 에픽으로 구성
    - 스토리는 오전 수업에서 말했듯이 사용자의 시점에서 기능 1-2개를 의미함
    - 작업은 구체적으로 동작을 위해 작업해야할 내용을 작성
    - 에픽은 스토리의 모음집이다.

- 컴포넌트
    - 컴포넌트는 작업을 하는 단위의 분류이다. 즉, FE, BE와 같이 간단하게 나눌 수도 있고 현업에서는 팀단위로 작성하는 경우도 있다.

### JQL

- Jira Query Language
- 이슈를 구조적으로 검색하기 위해 제공하는 언어
- SQL과 비슷한 문법을 가졌으나, Jira의 필드에 맞는 특수한 예약어를 제공한다
- 쌓인 Issue를 재가공해 유의미한 데이터를 도출해내는것에 활용한다

→ Search에서 Advanced를 눌러서 검색

operators

- =, ! =, >, > =
- in, not in
- ~(contains), !~(not contains)
- is empty, is not empty, is null, is not null

### JQL Dates

→ ~일 전 : -1d / ~주 전 : -1w

→ ~일 후 : 1d / ~주 후 : 1w

→ 비슷한 논리로 월, 년도 가능하다

### JQL Functions

- endOfDay(), startOfDay()
- endOfWeek() (토요일), startOfWeek() (일요일)
- endOfMonth(), startOfMonth(), endOfYear(), startOfYear()
- currentUser() → 현재 사용한 유저를 기준으로 쿼리를 작성할 때 사용

→ 추가로 계산을 하고 싶다면, 이것이 함수인 것임을 확인하여 괄호 안에 파리미터로 적으면 된다. 

※시간상으로 JQL을 학습할 시간이 없다면 기본 검색으로 선택한 이후 이를 JQL로 변환한다면 효과적인 학습이 가능하다
